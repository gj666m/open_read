# 需求分析启动模板

> 用途：每次收到新需求时，用此模板引导分析过程
> 配合使用：keyfield/分析报告输出规范.md（最终输出格式）

---

## 一、启动方式

在 Claude Code 中打开本项目后，发送：

```
我有一个新需求要分析。
[粘贴需求内容 / 说明需求来源 / 描述需求背景]
```

如果需求已存放在文件中：

```
读 data/需求信息/需求信息.md（或 data/需求原始资料/ 下的文件），
帮我分析这个需求。
```

---

## 二、分析流程

### Phase 1：需求接收与复述

**目标**：确保正确理解需求方在说什么。

执行步骤：
1. 用自己的话复述需求（一段话总结）
2. 列出需求中明确提到的要点
3. 标记需求中模糊、矛盾或缺失的部分

输出格式：

```
### 需求复述
[一段话总结]

### 明确要点
1. ...
2. ...

### 待澄清问题
1. [问题1] — 如果不确认，可能导致 [后果]
2. [问题2] — ...
```

**复杂度初判**：根据待澄清问题的数量和深度，判断需求属于简单/中等/复杂。

- 如果待澄清问题 <= 2 个且都很简单 → 跳过 Phase 2，直接进入 Phase 3
- 如果有 3 个以上需要确认的点 → 进入 Phase 2 深度澄清

---

### Phase 2：深度澄清（多轮对话）

**目标**：逐个确认 Phase 1 中标记的模糊点。

对每个待澄清问题：
1. 提出 2-3 种可能的解读（不要只问"这是什么意思"）
2. 每个解读说明影响范围（涉及哪些模块/逻辑/数据）
3. 等用户确认或补充

交互示例：

```
Claude: 关于"[模糊点]"，有两种可能的理解：
  A. [解读A]
  B. [解读B]

  如果是A，会影响：[影响范围]
  如果是B，会影响：[影响范围]

  你确认是哪种？

用户: [确认]

Claude: 明确了。记录：[确认结果]。
```

**退出条件**：所有待澄清问题都已确认，无剩余歧义。

---

### Phase 3：技术方案设计

**目标**：将确认后的需求转化为技术语言。

输出以下内容：

#### 3.1 业务规则提炼
- 核心业务规则列表（用精确的条件和公式描述）
- 数据流向（输入 → 处理 → 输出）
- 边界条件与异常场景

#### 3.2 技术方案
- 需要对接的外部系统/API（如有）
- 核心数据结构设计
- 关键算法/逻辑设计
- 技术选型建议（列举方案和利弊，让用户选择）

#### 3.3 任务分解
- 按 Phase 拆分开发任务
- 每个 Phase 包含：目标、涉及文件/模块、验证方式
- 标注 Phase 间的依赖关系

**关键原则**：
- 技术选型时列举方案和利弊，最终决策由用户做
- 如果有多种实现路径，先讨论再定稿
- 对接外部 API 时，先确认接口文档和真实返回格式

---

### Phase 4：输出确认

**目标**：生成最终分析报告。

执行步骤：
1. 按 `keyfield/分析报告输出规范.md` 的格式生成完整报告
2. 存放到 `data/output/YYYY-MM-DD_需求简称.md`
3. 请用户确认或修改

确认后的报告就是最终产物，用户将其中的「第6章 项目启动指令」复制到目标项目的 Claude Code 会话中使用。

---

## 三、复杂度自适应

### 简单需求（1-2 天的工作量）
- Phase 1 + Phase 3 合并（直接出方案）
- 跳过 Phase 2
- 输出报告使用简化模板

### 中等需求（1-2 周）
- 完整四阶段流程
- 标准输出模板

### 复杂需求（1 月+）
- Phase 2 可能需要多轮、跨天对话
- 输出报告使用完整模板
- 建议拆分为子需求分别分析

### 判断标准

| 特征 | 简单 | 中等 | 复杂 |
|------|------|------|------|
| 预估工作量 | 1-2天 | 1-2周 | 1月+ |
| 涉及系统 | 单系统 | 1-2个系统 | 3+系统 |
| 待澄清点数量 | <=2 | 3-5 | 5+ |
| 影响范围 | 单模块 | 多模块 | 跨项目 |

---

## 四、与下游项目的衔接

分析报告完成后，使用以下方式将成果传递到目标项目：

**方式一：复制启动指令**

打开目标项目的 Claude Code，粘贴报告中的「第6章 项目启动指令」内容。
目标项目的 Claude Code 会按 `keyfield/项目启动提示词模版.md` 的流程建立文档体系并开始开发。

**方式二：指定文件路径**

如果目标项目和本项目在同一台机器上：

```
读 /path/to/Demand-Flow_AI/data/output/YYYY-MM-DD_需求简称.md
这是需求分析的输出，请基于这份报告建立项目文档体系。
```

---

## 五、注意事项

1. **不要写代码**：这个项目的职责是分析需求，不是写代码。即使用户要求写代码，也应该提醒"这里是分析工作台，请到目标项目中执行"
2. **不要假设**：所有模糊点都必须跟用户确认，宁可多问一句也不要猜错方向
3. **保持中立**：技术选型时客观分析利弊，不偏向某个方案
4. **记录过程**：分析过程中的关键决策（为什么选A不选B）要记录在报告中
