# Karpathy LLM 编码准则

> 来源：forrestchang/andrej-karpathy-skills (GitHub)
> 基于 Andrej Karpathy 对 LLM 编码常见陷阱的观察总结
> 权衡：这些准则偏向谨慎而非速度。对于简单任务，自行判断。

---

## 1. 编码前先思考

**不要假设，不要隐藏困惑，揭示权衡。**

实现前：
- 明确陈述假设，不确定就问
- 多种理解时全部呈现，而非静默选择
- 有更简单的方案就说出来，必要时反驳
- 不清楚就停下来，说清楚困惑再问

## 2. 简单优先

**解决问题所需的最少代码，不做任何猜测性设计。**

- 不添加未要求的功能
- 不为单次使用代码做抽象
- 不添加未要求的"灵活性"或"可配置性"
- 不为不可能的场景做错误处理
- 如果写了200行但50行就能搞定，重写

自问："高级工程师会觉得这太复杂了吗？" 如果是，简化。

## 3. 精准变更

**只动必须动的，只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你会用不同方式
- 发现无关死代码，提一下但不要删

当你的变更产生孤立代码时：
- 删除你的变更导致的未使用的 import/变量/函数
- 不要删除之前就存在的死代码，除非被要求

**测试标准：每一行变更都应能追溯到用户的请求。**

## 4. 目标驱动执行

**定义成功标准，循环直到验证通过。**

将任务转化为可验证目标：
- "加验证" → "为无效输入写测试，然后让测试通过"
- "修bug" → "写一个能复现bug的测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

多步骤任务，列一个简短计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强成功标准让你可以独立循环。弱标准（"让它能用"）需要不断澄清。

---

## 有效性判断

这些准则有效的标志是：
- diff 中不必要的变更更少
- 因过度复杂导致的重写更少
- 澄清问题在实现之前就提出了（而不是犯错之后）
