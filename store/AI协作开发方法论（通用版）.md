# 用 AI 做开发：实战方法论

> 提炼自多个项目的实战经验
> 目的：作为启动项目时的操作指南，让 AI 辅助开发过程更顺畅、产出更可靠
> 适用于：Claude Code / Cursor / 任何 AI 编程助手

---

## 一、项目启动：建立「AI 上下文基础设施」

### 核心原则：AI 没有记忆，你的文档就是它的记忆

每个新 Session，AI 都是白纸。**上下文恢复的质量直接决定了 AI 的工作质量**。

### 建什么

| 文档 | 作用 | 维护时机 |
|------|------|---------|
| 项目进度总结 | 整体状态、对接情况、已知问题 | 每次重大改动后更新 |
| AI_review.md | 按 Session 记录推理过程、决策、踩坑 | 每次 Session 结束时记录 |
| 关键文件索引 | 哪个文件干什么，快速定位 | 新增/删除文件时更新 |
| CLAUDE.md（项目根目录） | 给 AI 的常驻指令（编码规范、偏好） | 一次性设置 |
| MEMORY.md（自动记忆） | 跨会话持久化的关键信息 | 稳定结论才写入 |

### 怎么用

**每次新 Session 的第一句话**：

```
读 keyfield/项目进度总结.md + keyfield/AI_review.md（重点看最后 3 个 Session）
```

目的：用最少的 Token 让 AI 恢复到与上次 Session 结束时相同的知识状态。

> 先花 2 分钟读文档恢复上下文，后面能省 2 小时排错。

## 二、需求沟通：先验证再动手

### 核心原则：AI 最擅长的是「执行明确的指令」，不是「理解模糊的意图」

### 方法论：「分步确认法」

**先讨论方案，不写代码** → 产出方案文档（明确了改动范围、旧逻辑删除、新逻辑添加、边界 case）→ 确认后再按方案执行。

### 实操建议

```
# ❌ 错误方式
"帮我把 XX 逻辑改一下，用新的字段来筛选"

# ✅ 正确方式
"我有一个需求想改造，先别动手，帮我分析一下方案"
 → [讨论完方案]
 → "方案确认，按这个文档执行：keyfield/xxx 方案.md"
```

### 判断什么时候需要「先讨论」

| 特征 | 直接动手 | 先讨论方案 |
|------|---------|-----------|
| 改动范围 | 1-2 个文件 | 3 个以上文件 |
| 逻辑确定性 | 明确知道怎么改 | 有多种实现路径 |
| 影响面 | 局部修改 | 涉及核心业务流程 |
| 可逆性 | 容易回滚 | 改了很难回头 |

## 三、验证策略：每个里程碑都要验证

### 核心原则：不要相信代码「应该没问题」，跑一遍才知道

### 验证层次

```
第一层：单元/接口验证
  → 每个接口/函数单独测试，打印原始输入输出
  → 典型错误：假设外部文档是对的（实际经常不一致）

第二层：业务逻辑验证
  → 用已知数据（手工模版/标准答案）对比脚本输出
  → 这是最快速确认逻辑正确性的方式

第三层：端到端验证
  → 从数据输入 → 处理 → 输出的完整链路
  → 最接近真实使用的验证

第四层：部署验证
  → 本地 OK 不代表服务器 OK
  → 环境差异、配置差异、缓存问题
```

### 验证的黄金法则

```
能用真实数据验证的，绝不用 mock 数据
能用完整流程验证的，绝不用单元测试替代
能让人工核对的，绝不相信脚本输出
```

## 四、排错方法论：从现象到根因

### 核心原则：AI 排错的瓶颈不是「不会修」，是「不知道哪里坏了」

### 通用排错流程

```
Step 1: 复现问题（确定现象）
  → 打印原始数据，看 AI 看到的是什么

Step 2: 隔离问题（缩小范围）
  → 是数据问题？逻辑问题？还是传递问题？
  → 方法：在每一层的输入输出打印数据，找到"正确→错误"的转折点

Step 3: 定位根因（找到原因）
  → 不要假设，用数据说话
  → 打开原始数据/API 返回值/数据库记录，逐条对比

Step 4: 修复并验证
  → 修复后用同样的测试用例验证
  → 同时检查是否有同类问题
```

### 常见排错陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 现象是间歇性的 | 重点关注数据中是否包含特殊字符（编码问题、格式问题） |
| 数据为空但逻辑看着没问题 | 检查调用方是否传了正确的参数，不要去改被调用函数 |
| 数据存在但查询不到 | 首先怀疑查询条件的数据格式是否一致 |
| 修改了代码但看不到效果 | 检查缓存、检查是否改对了文件、检查部署状态 |

### 给 AI 的排错指令模板

```
"我发现 [现象]。请帮我排查：
1. 先读 [相关文件]
2. 打印 [关键变量] 的值
3. 逐步分析从 [输入] 到 [输出] 的数据流
4. 找到数据从正确变成错误的转折点"
```

## 五、架构决策：让 AI 给你方案，你做选择

### 核心原则：AI 擅长列举方案和分析利弊，最终决策由你做

### 实操建议

```
# ✅ 好的提问方式
"我要做一个 XX 功能的页面，有 A 和 B 两种方案，
A 的优点是 xxx 缺点是 xxx，B 的优点是 xxx 缺点是 xxx，
你觉得哪个更适合？为什么？"

# ❌ 差的提问方式
"帮我做一个页面"
```

**关键追问**：
1. "这个方案有什么风险？"
2. "如果后面需求变了，改起来难不难？"
3. "有没有更简单的方案？"

## 六、增量开发：分 Phase 推进

### 核心原则：大改动拆成小步骤，每步可验证

### Phase 拆分要点

1. 每个 Phase 独立可验证
2. Phase 之间有明确的依赖关系
3. 即使在某个 Phase 中断，已完成的部分也是可用的
4. 出问题时范围锁定在一个 Phase 内

### 实操建议

```
# 当 AI 要一次改很多文件时：
"这个改动太大了，帮我拆成几个 Phase，
每个 Phase 完成后我可以验证一下，
确认没问题再进入下一个 Phase"
```

### 反面案例：一次性大改造

一次性新增/修改 10+ 个文件，如果有问题，排查范围巨大，不像 Phase 拆分那样能快速定位。

## 七、文档管理：Session 间信息传递

### 核心原则：每个 Session 结束时的文档更新 = 下个 Session 的启动效率

### 文档体系

```
AI_review.md          → Session 级别的详细记录
                        记录：推理过程、决策、踩坑、修改的文件
                        用途：下个 Session 恢复上下文

项目进度总结.md        → 项目级别的全局状态
                        记录：对接情况、已完成验证、已知问题、关键文件索引
                        用途：新 Session 快速了解项目全貌

MEMORY.md（自动）     → 跨项目记忆
                        记录：架构、偏好、踩坑记录
                        用途：自动注入每个 Session 的上下文
```

### 什么值得记录

| 记录 | 不记录 |
|------|--------|
| 关键决策及理由 | 临时的调试过程 |
| 踩坑和根因 | 修复中间的多次尝试 |
| 外部接口的真实格式 | 代码中已有的注释 |
| 业务规则确认 | 显而易见的逻辑 |
| 文件路径和功能 | 每次都变的内容 |

### Session 结束时给 AI 的指令

```
"帮我更新以下文档：
1. keyfield/AI_review.md — 新增本次 Session 的记录
2. keyfield/项目进度总结.md — 更新已完成/待办项"
```

## 八、部署和运维：从开发到生产

### 核心原则：本地跑通 ≠ 生产可用

### 部署经验

| 经验 | 详情 |
|------|------|
| 部署要早做 | 不要等项目做完才部署，尽早暴露环境问题 |
| 敏感配置不入 git | 配置文件手动传 |
| 前端缓存问题 | 修改 CSS/JS 后加版本号参数，强制刷新确认 |

## 九、与需求方的协作模式

### 核心原则：你是 AI 和需求方之间的翻译器

```
需求方
    ↓ 提供原始需求、业务规则、参考材料
你（开发者 + AI 操作者）
    ↓ 翻译为技术需求，让 AI 实现
AI
    ↓ 生成代码，输出验证数据
你
    ↓ 核对结果，确认或打回
需求方
    ↓ 验收，提出下一轮需求
```

**拿到需求后的第一步不是让 AI 写代码，而是**：
1. 把需求翻译成技术要点（字段、条件、规则）
2. 列出你不确定的点
3. 回去问需求方确认
4. 确认后再让 AI 动手

## 十、速查表：不同场景下的最佳指令

| 场景 | 指令 |
|------|------|
| **项目启动** | `读 keyfield/项目进度总结.md + keyfield/AI_review.md（最后 3 个 Session）` |
| **新功能开发** | `先别动手，帮我分析实现方案` → 讨论完 → `分 Phase 执行` |
| **Bug 排查** | `先定位根因，不要急着修复` |
| **需求对接后** | `需求方确认了 [要点]，帮我翻译成技术方案：需要改哪些文件、每个文件改什么` |
| **Session 结束** | `更新项目文档（进度 + review + 问题清单）` |
| **部署更新** | `帮我推代码，提醒我部署步骤` |

---

## 总结：AI 辅助开发的 5 条黄金法则

1. **先恢复上下文，再开始工作** — 每次新 Session 读文档
2. **先讨论方案，再写代码** — 大改动必有方案文档
3. **每步都验证，不要攒到最后** — Phase 拆分 + 逐步验证
4. **用真实数据验证，不用 mock** — 手工模版/API 返回值是最好的测试用例
5. **Session 结束时更新文档** — 下次启动的效率取决于这次的记录质量
